SPRING BOOT REST API DESIGN BEST PRACTICES

===============================
SECTION 1: PROFESSIONAL API DESIGN (VIDEO POINTS)
===============================

1. Resource-based URLs (No verbs)
Summary: Use nouns in URLs and HTTP methods for actions.
Best Practice: GET /users, POST /users
Bad Practice: /getUsers, /createUser

2. Proper HTTP Status Codes
Summary: Return meaningful HTTP status codes.
Best Practice: 201 Created, 404 Not Found
Bad Practice: Always returning 200 OK

3. Consistent JSON Structure
Summary: Use consistent naming convention.
Best Practice: camelCase everywhere
Bad Practice: mix of camelCase, snake_case

4. API Versioning
Summary: Version APIs to avoid breaking changes.
Best Practice: /api/v1/users
Bad Practice: /users (no versioning)

5. Pagination
Summary: Return data in chunks.
Best Practice: /users?page=1&size=10
Bad Practice: returning all records

6. Authentication vs Authorization
Summary: Separate identity and permission checks.
Best Practice: JWT + role-based access
Bad Practice: mixing auth logic everywhere

7. Structured Error Responses
Summary: Return detailed error objects.
Best Practice:
{
  "code": "USER_NOT_FOUND",
  "message": "User not found"
}
Bad Practice: "Something went wrong"

8. Filtering & Sorting
Summary: Use query params instead of multiple endpoints.
Best Practice: /users?status=active&sort=name
Bad Practice: /users/active/sortedByName

9. Security First
Summary: Build secure APIs.
Best Practice: HTTPS, validation, rate limiting
Bad Practice: exposing sensitive data

10. Design for Client, Not DB
Summary: API should not mirror DB tables.
Best Practice: custom DTO responses
Bad Practice: exposing entity structure

Best Practice: GET, POST, PUT, DELETE

Best Practice: UserResponse DTO

Best Practice:

Best Practice: @ControllerAdvice
Bad Practice: try-catch in every controller

16. Validation

Best Practice: @Valid, @NotNull
Bad Practice: no validation

17. Layered Architecture
Summary: Separate concerns.
Best Practice: Controller → Service → Repository
Bad Practice: business logic in controller

18. Documentation

Best Practice: Swagger/OpenAPI
Bad Practice: no documentation

19. Naming Conventions
Summary: Maintain consistency.

Bad Practice: inconsistent naming

20. Idempotency
